# Contributing

Thanks for taking the time to contribute. This document covers everything you need to get a working dev environment, make changes, and open a pull request.

## Prerequisites

| Tool | Version | Purpose |
|---|---|---|
| Go | 1.26.1+ | Build and test |
| Docker & Docker Compose | Any recent version | Local stack |
| golangci-lint | Latest | Linting |
| moq | Latest | Mock generation |

Install moq:

```bash
go install github.com/matryer/moq@latest
```

Install golangci-lint following the [official instructions](https://golangci-lint.run/welcome/install/).

## Local setup

```bash
git clone https://github.com/lays147/skip-the-line.git
cd skip-the-line
cp .env.example .env
```

Edit `.env` and fill in the three required variables:

```
GITHUB_WEBHOOK_SECRET=any-local-secret
GITHUB_TOKEN=<your-github-pat>
SLACK_BOT_TOKEN=<your-slack-bot-token>
```

Start the full stack:

```bash
make up
```

This starts the app, two HTTP echo servers (mock Slack on `:3000`, mock GitHub on `:4000`), and a Grafana OTel stack (UI at `http://localhost:3001`, admin/admin).

Send a sample webhook to your local instance:

```bash
make send-event EVENT=pull_request
```

## Running tests

Always use Makefile targets — never raw `go test` directly.

```bash
make test          # run all unit tests
make test-cover    # run tests + open HTML coverage report
```

To run a single test by name:

```bash
go test -run TestFunctionName ./internal/notification/
```

Tests follow these conventions:
- Table-driven with `t.Run` subtests
- Black-box packages (`package notification_test`) for public API; white-box (`package webhook`) when unexported helpers are needed
- OTel providers are replaced with noop implementations: `noop.NewMeterProvider()`, `noop.NewTracerProvider()`
- Mocks come from `internal/mocks` — never write them by hand

### Makefile targets

| Target | Description |
|---|---|
| `make up` | Start the full stack with Docker Compose |
| `make down` | Stop and remove the stack |
| `make logs` | Tail app logs |
| `make test` | Run all unit tests |
| `make test-cover` | Run tests and open an HTML coverage report |
| `make lint` | Run golangci-lint |
| `make build` | Build the binary locally |
| `make generate` | Regenerate moq mocks after an interface change |

## Linting

```bash
make lint
```

Fix any issues before opening a PR. The CI pipeline runs the same linter.

## Regenerating mocks

Mocks are generated by [moq](https://github.com/matryer/moq) into `internal/mocks/`. The `//go:generate` directives live next to the interfaces they mock (`recipients.go`, `service.go`).

After changing an interface, regenerate:

```bash
make generate
```

Never edit files inside `internal/mocks/` by hand — they will be overwritten.

## Adding a new event type

1. Add a handler method in `internal/notification/service.go`
2. Register the new event type/action in the routing switch
3. Add the corresponding interface method to `internal/notification/recipients.go` if a new external call is needed, then run `make generate`
4. Write table-driven tests in `internal/notification/service_test.go`
5. Add a sample payload under `scripts/` and wire it into `send-webhook.sh` so it can be tested with `make send-event`

## Project layout

```
cmd/server/          # main entrypoint — wires all dependencies
internal/
  config/            # environment variable loading
  github/            # GitHub API client
  health/            # /healthz and /readyz handlers
  metrics/           # OTel metrics setup
  mocks/             # generated moq mocks (do not edit)
  notification/      # core event routing and Slack dispatch
  slack/             # Slack API client
  subscription/      # subscriptions.yaml loader and Registry
  webhook/           # HTTP handler, HMAC validation, dedup cache
scripts/             # helper scripts (send-webhook.sh, sample payloads)
docs/images/         # screenshots used in README
```

## Pull request guidelines

- Keep PRs focused — one logical change per PR
- All new behaviour must have tests; aim to keep coverage above the current baseline
- Run `make test` and `make lint` locally before pushing
- Write a clear PR description explaining *why*, not just what changed
- Errors must be wrapped with `%v`; avoid wrapping errors within the same package
- Do not add comments unless the logic is non-obvious

## Reporting issues

Open an issue on GitHub with a clear description and, where applicable, the relevant log output and the webhook payload that triggered the problem.
